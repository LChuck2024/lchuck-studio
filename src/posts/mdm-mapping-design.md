在快消行业做了 10 年数据治理，我见过最多的坑不是技术 Bug，而是**业务口径不统一**。品类名称「洗发水」和「洗发液」被当成两个东西统计，客户编码 A 系统和 B 系统各有一套，报表永远对不上。本文复盘如何用「配置驱动映射表」解决这类主数据病灶，把硬编码的 `CASE WHEN` 变成可维护的 ETL 原子能力。

---

## 一、主数据病灶：缺乏 SSOT 的代价

**Single Source of Truth（SSOT）** 是主数据治理的基石。但在现实中，很多企业的数据流是这样的：

1. **同义词未归一化**：品类「洗发水」「洗发液」「洗护发」本质同一类，却被拆成 3 个维度统计。
2. **编码体系割裂**：ERP 用 8 位数字，CRM 用 6 位+字母，数仓要建一张「编码对照表」才能 JOIN。
3. **映射逻辑散落**：每个报表、每个 ETL 脚本里都有一大坨 `CASE WHEN` 或 `IF`，改一处漏十处。

**后果**：数据口径不一致 → 业务不信任报表 → 人工对账 → 效率崩塌。

---

## 二、破局思路：映射表 + 配置驱动

核心原则：**把「业务规则」从代码里抽出来，放进配置表。**

| 层级 | 原来 | 现在 |
|------|------|------|
| 映射逻辑 | 硬编码在 SQL/Python | 存在 `dim_xxx_mapping` 表 |
| 新增同义词 | 改代码、发版 | 插一行配置 |
| 审计追溯 | 靠 Git 历史 | 映射表带 `valid_from` / `valid_to` |

**映射表设计范式**（以品类为例）：

```sql
-- dim_category_mapping：源值 → 标准值
CREATE TABLE dim_category_mapping (
  source_value    VARCHAR(100),   -- 原始值（如「洗发液」）
  standard_value  VARCHAR(100),   -- 标准值（如「洗发水」）
  valid_from      DATE,
  valid_to        DATE,
  created_by      VARCHAR(50)
);
```

ETL 时只需 `LEFT JOIN dim_category_mapping`，用 `COALESCE(m.standard_value, raw.category)` 兜底未配置的脏数据。

---

## 三、与 ETL 引擎的衔接

映射表不是孤立的，它应该成为 ETL 流水线的一环：

1. **配置层**：维护 `dim_xxx_mapping`，可由业务或数据治理专员通过低代码界面维护。
2. **引擎层**：通用 ETL 引擎读取配置，在 `SELECT` 阶段自动注入 `JOIN mapping` 逻辑。
3. **监控层**：对 `source_value` 在映射表中无匹配的记录打标，进入「待治理队列」，避免静默丢失。

这样，**新增一个同义词 = 插一行配置**，无需改任何 ETL 代码。

---

## 四、中台化进阶：AI 辅助发现同义词

当映射表规模变大，人工维护会吃力。可以引入 **Fuzzy Matching** 或 **LLM**：

- **规则可覆盖的**：用编辑距离、拼音相似度等算法，自动推荐「洗发液 → 洗发水」这类候选。
- **规则难覆盖的**：把未匹配的 `source_value` 批量丢给 LLM，让它输出「最可能的 standard_value」建议，人工审核后入库。

这样，映射表从「人工维护」升级为「AI 辅助 + 人工兜底」的闭环。

---

## 五、小结

| 痛点 | 方案 |
|------|------|
| 同义词未归一化 | 建立 `dim_xxx_mapping` 配置表 |
| 逻辑散落、难维护 | 配置驱动，ETL 引擎统一 JOIN |
| 发现新同义词成本高 | Fuzzy Matching + LLM 辅助推荐 |

主数据治理不是一次性项目，而是**持续迭代的配置化能力**。把映射表设计好，后续的扩展成本会指数级下降。
